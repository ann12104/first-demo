# 小游戏开发方案与技术选型（Unity + C# + 无后端）

## 一、项目目标与边界

### 1.1 项目目标

- 基于现有企划，落地一款面向东南亚市场的单机猫舍养成小游戏；
- 采用 Unity + C# 客户端实现，所有核心数据本地存储；
- 2-4 周完成可上线版本（Google Play 首发）；
- 重点保证：低端机可运行、广告变现链路完整、玩法闭环稳定。

### 1.2 技术边界

- 无自建后端、无账号系统、无实时联机；
- 不依赖云存档，数据以本地 JSON 持久化；
- 伪社交通过系统分享行为触发，不做好友关系与回执系统；
- 排行榜采用本地模拟数据，不接入在线榜单服务。

## 二、整体技术架构

### 2.1 客户端分层

- 表现层（UI）：主界面、钓鱼界面、装扮界面、弹窗；
- 业务层（Domain）：猫咪成长、钓鱼产出、互助次数、广告奖励；
- 数据层（Data）：本地存档读写、配置加载、每日重置；
- 平台层（Platform）：广告 SDK、分享 SDK、权限与生命周期。

### 2.2 核心原则

- 单向数据驱动 UI：业务状态变化后刷新界面，避免界面直接改业务数据；
- 简易 MVC：View 只负责展示与交互转发，Controller 处理流程，Model 管状态与数据；
- 模块低耦合：Cat/Fishing/Ad/Share/Save 解耦，方便后续加活动模块；
- 配置化优先：掉落概率、奖励数值、广告冷却等放入配置文件；
- 可回归：关键功能（存档、重置、奖励发放）可快速验证。

## 三、技术选型与理由

### 3.1 引擎与语言

- Unity 2021 LTS（建议 2021.3.x）：稳定、生态成熟、移动端兼容好；
- C#：与 Unity 原生一致，开发效率与维护性高。

### 3.2 UI 方案

- 固定使用 UGUI（开发快、学习与维护成本低）；
- 采用简易 MVC 组织 UI：Panel(View) + Controller + UIModel；
- 页面级解耦：每个页面维护独立 Controller，避免单点超大脚本；
- 文本多语言统一走 Key 映射，不在 UI 预制体中写死文案。

### 3.3 资源与配置

- 配置格式：JSON（简洁直观，方便策划协作）；
- 图集方案：Sprite Atlas，降低 DrawCall；
- 动画方案：轻量序列帧 + 少量 Tween，控制 CPU/GPU 压力。

### 3.4 本地存档

- 存储路径：Application.persistentDataPath；
- 存档格式：JSON + 版本号（便于后续兼容）；
- 存档机制：关键操作后即时保存 + 退后台自动保存；
- 安全措施：基础字段校验与容错回退（坏档自动恢复默认存档）。

### 3.5 广告与分享

- 广告：AdMob + Pangle（激励视频优先，插屏限频，Banner 常驻底部）；
- 分享：移动端原生分享插件（Android/iOS），返回前台即判定“互助完成”；
- 合规：奖励必兑现，广告失败给出可重试路径。

### 3.6 分析埋点（无后端前提）

- 最小埋点接入 Firebase Analytics 或 Unity Analytics；
- 仅记录关键漏斗：新手完成、首次广告、次日启动、7日启动；
- 不采集个人敏感信息，满足隐私政策约束。

## 四、模块实现方案（可直接进入开发）

### 4.1 CatManager（猫咪养成）

- 管理状态：饥饿/干净/快乐、成长阶段、亲密度；
- 输入事件：喂食、梳毛、陪玩；
- 输出事件：状态变化、成长升级、专属动作解锁；
- 与 SaveManager 协同：每次成长或关键状态变化后保存。

### 4.2 FishingManager（钓鱼产出）

- 管理体力值、恢复计时、钓鱼结算；
- 按掉落配置计算鱼种与道具产出；
- 支持激励视频加成：满体力、双倍渔获、保底稀有；
- 与 UI 交互保证“一键钓鱼 + 短等待 + 明确反馈”。

### 4.3 ShareHelpManager（伪社交互助）

- 每日 3 次互助上限，本地日期重置；
- 调起分享后，应用回到前台即发放奖励；
- 防刷策略：次数上限 + 当日标记 + 最小触发间隔。

### 4.4 AdManager（广告变现）

- 封装激励视频/插屏/Banner 生命周期；
- 插屏限频（2-3 分钟）与场景白名单；
- 广告失败兜底：提示 + 重试，不阻断主流程。

### 4.5 SaveManager（本地持久化）

- 统一管理所有存档结构体；
- 启动时做版本迁移与默认值补全；
- 提供每日重置入口（互助次数、签到状态、免费宝箱）。

### 4.6 UIManager（界面路由）

- 负责界面切换、弹窗调度、红点提示；
- 订阅业务事件刷新显示，不直接写业务逻辑；
- 多语言动态切换与低分辨率适配。

## 六、开发排期（4 周）

### 第 1 周：核心玩法可跑通

- 完成猫咪养成、钓鱼、基础 UI、存档读写；
- 达成“钓鱼→养成→成长→保存”闭环。

### 第 2 周：商业化闭环

- 接入激励视频、插屏、Banner；
- 完成分享互助链路与每日次数重置；
- 增加关键埋点与基础运营参数配置。

### 第 3 周：留存模块与本地化

- 实现签到、宝箱、模拟排行榜；
- 上线英语/印尼语/越南语；
- 调整新手引导与数值体验。

### 第 4 周：优化与上架准备

- 性能优化（包体、内存、帧率）；
- 合规检查（隐私政策、广告提示、权限）；
- 打包提测，准备商店素材与灰度发布。

## 七、关键性能与质量标准

- 包体：Android 首包 ≤ 50MB；
- 帧率：低端机主场景稳定 30 FPS；
- 内存：常驻内存可控，避免频繁 GC 卡顿；
- 启动：冷启动目标 8 秒内进入主界面；
- 稳定性：核心链路（钓鱼、广告奖励、存档）无阻断性缺陷。

## 八、风险清单与应对

- 广告填充率波动：预置无广告兜底奖励策略，避免体验断层；
- 本地时间篡改：每日重置增加合理性校验，降低刷奖励收益；
- 多语言截断：UI 预留弹性布局，关键按钮做文本长度适配；
- 低端机卡顿：减少透明叠层与高频动画，控制图集尺寸。

## 九、MVP 验收标准

- 用户可完成 10 分钟内至少 3 次完整核心循环；
- 激励视频奖励成功率高且可感知；
- 分享互助每日次数控制准确；
- 退出重进后数据正确恢复；
- 三种语言可正常切换且无关键错译。

## 十、结论

在“Unity + C# + 无后端”约束下，该项目技术路径可行，且能在短周期内快速上线验证。建议按 MVP 先跑通核心闭环与广告变现，再通过活动与装扮内容持续迭代提升留存和收益。
